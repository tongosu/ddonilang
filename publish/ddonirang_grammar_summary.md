# 또니랑 문법 요약 (발표용, 주제별)

> 목적: “언어 발표”에서 **문법의 핵심만** 빠르게 설명하기 위한 요약입니다.

## 1) 문장과 종결
- 또니랑의 기본 단위는 **문장**입니다.
- 문장은 보통 `.` 으로 끝납니다.

## 2) 조사 경계 `~` (정본 표기)
- 한국어의 조사(을/를, 로/으로…)를 코드에 반영합니다.
- **정본(캐논) 표기**에서는 조사 경계를 `~` 로 드러내어 해석이 흔들리지 않게 합니다.
- 예: `문~을`, `말~로`, `주문~으로`, `나~는`

## 3) 상태 변경(대입): `<-`
- 상태를 바꾸는 표기는 **한 가지로 통일**합니다.
- 예: `누리.문_상태 <- "열림".`

## 4) 정의 vs 실행(호출)
- **정의**: `이름:움직씨 = { ... }.`
- **호출(실행)**: `(... ) 이름 하기.` 또는 `(... )~로 이름 하기.`

## 5) 움직씨(행동)와 매개(예)
- 움직씨는 “세계에 변화를 주는 행동”입니다.
- 예:

```ddn
(주문:말~로) 문~을_열기:움직씨 = {
    바탕.문_상태 <- "열림".
    (주문) 보여주기.
}.
```

## 6) 실행 꼬리(어미): `~기 / ~하기`
- 실행은 꼬리(어미)로 고정해, 사람이든 AI든 같은 의도를 같은 꼴로 쓰게 합니다.
- 예: `("열려라!")~로 문~을_열기 하기.`

## 7) 토막과 평가 표지(언제 실행되는지)
- `{ ... }` 블록은 “토막”으로 다루고, **평가 표지**로 실행 시점을 명확히 합니다.
- 발표에서는 개념만 소개하고, 자세한 규칙은 문서/README로 연결해도 좋습니다.

## 8) 연결(파이프) — `...해서 ...`
- 여러 행동을 이어 붙일 때 연결 방식이 하나로 수렴되도록 설계합니다.
- 목표: “추측/암묵 규칙”을 줄여 결정성을 강화.

## 9) 계약(조건) — `늘지켜보고 { ... }.`
- 움직씨가 실행되기 위한 **조건(계약)** 을 명시합니다.
- 조건 위반은 “조용히 무시”가 아니라 **진단/기록(거울)** 로 남기도록 목표를 둡니다.

## 10) 결정성(재현) = 문법
- 결정성은 엔진 옵션이 아니라 **문법 설계 목표**입니다.
- 조사 경계, 실행 꼬리, 단일 대입(`<-)`, 단일 연결(`해서`)처럼
  **모호함을 문법에서 먼저 제거**해 재현성을 확보합니다.
