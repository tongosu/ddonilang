## 핵심 원칙 (2026-07-06 수정 — 사용자 지적으로 정정)

**결정론 성역은 이야기(Iyagi) 하나뿐이다.** 샘/누리/거울/보개 네 계층은 전부 엔진 기계장치이며 어떤 언어(Rust/JS/WASM 등)로 구현되어도 무방하다. 이야기와 주고받는 데이터 계약 3개만 고정되면 된다:

1. **샘→이야기**: 입력 스냅샷 포맷(`InputSnapshot`).
2. **이야기→누리**: patch 포맷(`Patch` 구조체) — 이미 타입으로 강제되어 있어 안전(Q28 검증: 위반 없음).
3. **누리→거울/보개**: 관찰 포맷(`ObservationManifest`/`state_hash` 경계) — D40이 현재 미착륙임을 확인한 지점.

최초 초안은 "이야기**와 누리** 둘 다 DDN이어야 한다"고 잘못 썼다. 실제로 누리(patch 적용 + 해시 계산)는 사용자가 로직을 저작하는 자리가 아니라 고정된 기계적 병합 연산이며, 현재도 Rust로 구현되어 있다(`core/src/engine.rs:40-48`, Q28에서 실측 확인). DDN이 강제하는 장치(Fixed64, ambient I/O 없음, 정본 문법)가 필요한 이유는 "사용자가 실제로 로직을 쓰는 자리에서 결정론이 깨지지 않게 하기 위함"이고, 이 조건을 만족하는 유일한 계층이 이야기다. 나머지 계층은 사용자 로직이 없으므로 이 장치가 필요 없다 — 대신 계약(고정된 데이터 포맷)만 있으면 된다.

이것은 "DDN이 영원히 이야기만 만드는 언어냐"는 우려에 대한 답이기도 하다: 그렇다, DDN은 이야기 저작 전용 언어이고, 이건 한계가 아니라 설계다. 게임 엔진에서 로직은 스크립팅 언어(Lua/C#), 렌더러는 시스템 언어(C++)인 구조와 동일하다.

### 3단계 구분 (2026-07-06 추가 — "특정 게임 시나리오 DSL처럼 좁아지는 것 아니냐"는 지적 반영)

위 "이야기만 DDN, 나머지는 계약"은 2단계로 너무 납작하다. 실제로는 3단계다:

1. **결정론 저작(이야기)** — DDN 전용, 대체 불가. 모델 로직.
2. **비결정론이지만 DDN으로 기술 가능한 저작 표면** — 보개/슬기에게 "뭘 보여줘/뭘 물어봐"를 선언하는 층. 계속 성장해야 하는 지점.
3. **엔진 기계장치(원시 접점)** — 실제 픽셀 칠하기, 실제 LLM API 호출, 실제 OS 이벤트 캡처. 영원히 네이티브 — 모든 언어의 공통 한계이지 DDN만의 제약이 아니다.

2번은 이미 SSOT에 실체가 있다:
- **`모양 {}`**(`SSOT_LANG_v24.12.9.md:7749`): primitive-only 특수 블록, partial 구현. 결과값(`모양값`)은 **view-only**로 명시되어 `state_hash`에서 제외된다. `모양씨`(재사용 생성기), `겹보기씨`(오버레이 레이어)까지 이름이 있다. 보개에게 "뭘 그릴지" 선언하는 문법이 이미 DDN 안에 있고, 실제로 칠하는 건 네이티브 — 정확히 이 3단계 구조다.
- **`슬기.의도{}`**(`SSOT_LANG_v24.12.9.md:129`): "슬기에게 뭘 요청할지"를 선언하는 블록으로 이미 제안된 적이 있으나 "current-line core grammar로 추가하지 않는다"고 보류된 상태다. 폐기가 아니라 보류다.
- **`OI-BOGAE-SECTIONS-01`**(`SSOT_OPEN_ISSUES_v24.12.9.md:1705`): "보개/슬기/샘을 문법 블록으로 둘지 vs 길잡이말로만 둘지"가 이미 미결 이슈로 등록되어 있다.

**Claude 추천:** DDN이 "렌더러/LLM 클라이언트를 직접 짤 수 있는 범용 언어"가 될 필요는 없다(정체성 희석 + 저효익). 대신 2번 저작 표면(보개/슬기에게 뭘 시킬지 선언하는 문법)을 계속 키우는 게 맞는 방향이며, 이미 길이 나 있다 — `모양{}`은 AGE4에서 완성 진행, `슬기.의도{}`는 보류를 재검토할 시점(당시보다 슬기 역할이 구체화됨)이라고 본다.

## D39 — 결정론 성역 공식화 (이야기 — 누리 아님, 정정됨)

**설명:** 현재 DR-152/DR-178은 "state_hash에 무엇이 포함되는가"를 규정하지만, "어느 계층이 사용자 저작 로직을 실행할 수 있는가"는 규정하지 않는다. 사용자/AI가 실제로 로직을 저작하는 유일한 지점은 **이야기(Iyagi)** 뿐이다(가지, lesson). 누리(Nuri)는 이야기가 만든 patch를 기계적으로 world에 병합하고 해시를 계산하는 고정 엔진 배관이며, 사용자 로직이 들어가는 자리가 아니다 — 실제로 지금도 Rust로 구현되어 있다(Q28 검증: `core/src/engine.rs:40-48`). 이걸 명문화해두지 않으면 나중에 "성능을 위해 네이티브 확장 포인트를 이야기 계층에 추가하자"는 제안이 나왔을 때 막을 SSOT 근거가 없다.

**신규 제안 문구(안):**
> 이야기(Iyagi) 계층 — 즉 가지/lesson 등 사용자·AI가 저작하는 모든 모델 로직 — 은 오직 또니랑(DDN) 언어로 작성된 코드여야 한다. 네이티브 플러그인, 동적 라이브러리 로드, 임의 스크립팅 훅은 이 계층에 들어올 수 없다. 샘/누리/거울/보개 네 계층은 사용자 저작 로직이 없는 엔진 기계장치이므로 이 요구사항의 대상이 아니다 — 이들은 이야기와 주고받는 데이터 계약(입력 스냅샷/patch/관찰 매니페스트)만 지키면 어떤 언어로 구현되어도 무방하다. 성능 최적화는 언어 처리기(파서/정본화기/런타임) 내부 구현에서만 이뤄진다.

**Claude 추천:** 채택. 비용이 거의 없고(현재 동작을 문서화할 뿐), 나중에 압박이 왔을 때 확실한 방어선이 된다.

**결정 필요:** 이 문구를 SSOT_MASTER의 어느 절에 반영할지(신규 절 vs 기존 아키텍처 절 보강)는 사용자가 SSOT 반영 시 정할 사항. 지금은 승인/보류만 결정하면 됨.

## D40 — 관찰자 계약 (보개 SDK + 거울 소비 도구 통합)

**설명:** 보개(Bogae)는 이미 "read-only 관찰/렌더"로 정의되어 있고, AGE2 로드맵에 `ObservationManifest`(엔진 네이티브 생성, 현재는 UI 합성이 fallback — `SSOT_MASTER_v24.12.9.md:1253,1283`)가 계획되어 있다. 즉 "무엇을 관찰할 수 있는가"를 엔진이 선언하는 개념 자체는 이미 SSOT에 있다. 지금 없는 건 "그 관찰 대상을 소비해서 임의의(비결정론적) 렌더러/분석 도구를 만드는 쪽"의 계약이다.

거울(Geoul)이 기록한 로그(사후 분석 도구)도 성격이 같다 — 이미 결정론적으로 박제된 사실을 읽기만 하므로, 소비하는 도구 쪽이 아무리 비결정론적이어도(랜덤 색상, AI 요약, 애니메이션 타이밍) 원본에 영향이 없다.

**신규 제안 문구(안):**
> **관찰자 계약(Observer Contract):** 보개 확장(렌더러/UI 패널)과 거울 로그 소비 도구는 다음을 만족하면 임의로(비결정론적으로) 저작될 수 있다.
> 1. `ObservationManifest`(또는 거울 기록 파일)로 명시적으로 선언된 대상만 읽는다 — 임의 내부 상태 직접 접근 금지.
> 2. 누리(Nuri) world에 쓰기 접근을 하지 않는다(엔진 mutation 함수 호출 금지).
> 3. 확장의 내부 로직(렌더링 타이밍, 색상 난수, 3rd-party 라이브러리 사용 등)은 `state_hash`/`trace_hash`에 어떤 영향도 주지 않는다.
> 이 3원칙을 만족하는 한, 보개 확장은 결정론 요구사항 밖에 있다 — 이것이 "가지/보개 직접 설계(결정론 없는 버전)" 트랙의 공식 근거다.

**Claude 추천:** 채택, 우선순위 최상위. Q26이 실제로 겪은 문제(마운트 관례 베끼기)를 해결하는 가장 직접적인 항목. 단, SDK 코드 자체는 지금 만들지 않는다 — 문서화만 지금 하고, 실제 구현은 AGE2 `ObservationManifest` 엔진 네이티브 경로가 고정된 뒤 진행(이미 SSOT 로드맵에 순서가 있음).

**결정 필요:** 이 3원칙 문구가 맞는지, 특히 "거울 소비 도구"를 보개와 같은 계약으로 묶는 것에 동의하는지.

**중요한 구분(오독 방지):** "비결정론이 허용된다"(D40)와 "영원히 DDN이 아닌 언어로 짜야 한다"는 별개의 축이다. D40이 고정하는 건 전자(결정론 요구사항 면제)뿐이고, 후자를 강제하지 않는다. 실제로는:
- 보개가 **무엇을 보여줄지**(관찰 대상 선택, 레이아웃, 데이터 매핑) 저작하는 층은 원리상 DDN 표현으로 넓어질 수 있다 — AGE4 로드맵의 `모양{}`이 정확히 이 방향(선언적 시각 블록)이다.
- 실제 픽셀/DOM을 **칠하는** 최하단 렌더 엔진만 항상 네이티브(JS/WASM)로 남는다 — 이건 브라우저 자체가 C++로 만들어졌다고 HTML/CSS 저작자가 "갇혀 있다"고 하지 않는 것과 같은 구조다.
- 샘도 마찬가지로 OS 이벤트 캡처(네이티브)와 "6개 입력원천으로 정규화하는 규칙"(DDN 표현 가능성 있음)을 구분해야 한다.
이 문서의 D40/D41은 "엔진의 raw 접점은 좁고 고정된 네이티브 계약으로 남는다"는 것만 못박는다 — "저작 표면이 영원히 DDN 밖에 있다"는 주장이 아니다. 저작 표면 확장은 커널 로드맵(모양{} 등)의 별도 트랙이다.

## D41 — 확장 입력 계약 (샘 + 슬기 통합)

**설명:** 이미 SSOT_MASTER `외부개입 공통 경계 — MUST`(L835-849)에 원칙이 있다: 외부개입은 세계 상태를 직접 수정하지 않고, 반드시 샘으로 정규화되며, 거울에 기록 가능해야 한다. 입력원천 6값(`사람/슬기/밖일/일정/이어전달/펼침실행`)도 이미 잠겨 있다. 즉 **D41은 새 규칙이 아니라 기존 MUST-절을 "미래의 신규 입력원천에도 동일하게 적용된다"고 명시적으로 확인하는 것**이다 — 예를 들어 게임 컨트롤러, 네트워크 멀티플레이어 입력, 외부 센서 데이터 같은 것을 나중에 추가하더라도 이 6값 스키마의 확장으로 다뤄야지, 별도 특권 경로를 만들면 안 된다.

슬기(Seulgi)는 이미 이 6값 중 하나(`슬기`)로 들어와 있으므로, "슬기가 제안한 행동이 실제로 반영되려면 항상 샘을 거친다"는 건 새 결정이 아니라 이 계약의 인스턴스임을 재확인하는 것뿐이다.

**신규 제안 문구(안):**
> 향후 추가되는 모든 신규 입력원천(게임 컨트롤러, 네트워크, 센서 등)은 기존 `입력원천` 6값 스키마의 확장으로 편입되어야 하며, `외부개입 공통 경계 — MUST`(직접 상태 변경 금지/샘 경유/거울 기록 가능)를 동일하게 따른다. 슬기(AI 어드바이저)를 포함한 어떤 입력원천도 이 경계를 우회하는 별도 경로를 가질 수 없다.

**Claude 추천:** 채택하되 "재확인" 성격임을 명확히. 실질적으로 이미 SSOT에 있는 내용이라 리스크가 거의 없다.

**결정 필요:** 이 재확인 문구를 신규 D-번호로 남길지, 아니면 그냥 기존 MUST-절 옆에 각주로 붙일지(별도 결정번호가 필요 없을 수도 있음).

## Codex 검증 위임 (실행 전 필수)

3. 이야기/누리 계층 실행 경로에 DDN 언어가 아닌 네이티브 훅/플러그인 로드 지점이 실제로 있는지(D39 위반 여부).
